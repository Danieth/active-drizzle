## Feasibility verdict (research, 2026-07-18)

The RFC is buildable. Three findings from the code that de-risk it:

1. **`permit` fn form already exists** — `permit?: string[] | ((ctx, ctrl) => string[])` in
   `packages/controller/src/metadata.ts:54`, resolved at runtime in `buildPermittedData`
   (`crud-handlers.ts:224`). And in `defaultUpdate` the record is loaded (line 178) **before**
   permit resolution (line 180) — so record-aware permit is a parameter-threading change,
   not a control-flow change.
2. **The extraction machinery is already the right shape** — ts-morph walks model classes
   (`extractor.ts`), validator bodies are lifted verbatim into generated Clients
   (`react-generator.ts:271`), and the dep-inference engine (`validation-deps.ts`) is a
   receiver-parametrizable walker. Extracting `label/help/copy/presentIf/...` from Attr
   config objects is the same trick on ObjectLiteralExpressions. `presentIf: d => ...`
   needs the walker generalized from `this` to an arbitrary root param — small refactor.
3. **Projection filtering already runs** — `controllerProjectionFields()` +
   `depsFitProjection()` gate which validators ship per controller Client
   (`react-generator.ts:241–251`). `expose` slots in as a redefinition of that set, not
   a new mechanism.

**Honest scope note:** RFC M0–M6 in full is multi-week. The weekend cut line is drawn so
that every phase above it ships something usable and tested; everything below is stretch.

## Phase 2 — Form surface: M3 minimal (Sunday morning) (`react`)

### API v2 — LOCKED (2026-07-18): no `f`, no `Present`

Naming law (codegen-enforced): **camelCase = fields** (`loan.amount`); **PascalCase =
components** (`loan.Form`, `loan.Submit` — column named `Form` ⇒ codegen error + alias hint);
**`$` = framework scalars** (`loan.$status`, `loan.$errors`, `loan.$dirty`, `loan.$submit()`,
`loan.$draft`) — because `status` is too common a column to squat on. JSX member expressions
need no capitalization, so the field handle IS the component (callable + `.error`/`.meta`).
`edit` prop: absent = view · `edit` (bool) = editable w/ Attr's default presenter ·
`edit="name"` = named override. `edit` never inferred (wall #2 intact). Two hooks
(`useEditForm`/`useNewForm`), ONE Handle type, identical JSX — mode lives in the hook.

```tsx
<loan.Form>
  <loan.amount edit />
  <loan.targetRate edit="bpsSlider" label="Target (bps)" />
  <loan.status />
  <loan.assets>{asset => (<><asset.name edit /><asset.value edit /><asset.Remove /></>)}</loan.assets>
  <loan.assets.Add defaults={{ value: 0 }}>Add asset</loan.assets.Add>
  <loan.Submit>Save</loan.Submit>
</loan.Form>
```

Presenter contract (the plugin layer): registry = declaration-merged interface (module
augmentation = how UI-kit packages plug in) + runtime `registerPresenter(name, { kind,
requires?, component })`. Presenter receives `{ value, bind, meta, overrides, draft, mode }`:
`meta` from the backend Attr (label/help/info/copy resolved per discriminant/constraints/kind),
call-site `overrides` win, `draft` = the ENTIRE projected object — safe by construction
(unexposed fields don't exist on the type and never crossed the wire). **`requires` gate:**
presenter declares meta it needs ⇒ using it on a field lacking that meta = compile error
(per-field meta types from codegen) + dev-mode runtime backstop.

### Auth — LOCKED (2026-07-18): adapter in, verdict out

**Authentication is not AD's problem.** Contract = three sentences: something upstream turns
a request into `ctx`; the app defines `AppContext`; AD never looks inside it. Clerk = ~20
lines of app-side Hono middleware in front of the existing adapter — swap providers by
rewriting those lines, zero AD/controller changes (controllers consume `ctx.userId`, never
`clerk.*`). **Authorization is already built:** `scopeBy` (rows) · `expose` (read) ·
`permit(ctx, record)` (write) · `abilities`/`can` (shipped verdict) · role-in-model ban.
**Presenters never receive roles** — `abilities` IS the auth status (server-computed, drift-
proof); presenters are app components and may use the app's own auth context (Clerk hooks)
for *decoration* only (e.g. "upgrade to edit" copy). Enforcement from the verdict,
decoration from context, never the reverse. Docs task: "Auth" page under Controllers
(adapter recipe: Clerk / Auth.js / test-fake) → add to Phase 4.

### Error architecture — LOCKED (2026-07-18)

One recursive `ErrorBag` per FormSession: `{ fields, base, children }`. Sources funneled:
client `validate()` ∪ server 422 (deduped) · re-fielded invisible-field issues · transition
guard failures · 409 text. Nested routing by identity (`id` / `_key` echo), never index.
Uniform API at every depth: `handle.field.errors` / `bind.errors` / `handle.$errors.any|count`
/ array rollup `loan.assets.$errors` / `<BaseErrors/>`. Presenters are depth-blind.

## Design decisions (locked / to lock)

1. **Projection = `expose` (data availability), not permit (editability).** A validator
   reading `status` (view-only) + `amount` (editable) SHOULD run client-side — the client
   has the data. `deps ⊆ expose` is the fail-closed boundary. This *changes* today's
   permit∪include projection — deliberate, migrate in 1d.1.
2. **`permit` signature breaks to `(ctx, record)`.** Pre-1.0; RFC form is right; `ctrl`
   reachable via closure where needed. Changelog note.
3. **No TanStack Form — LOCKED (2026-07-18).** The form problem is two layers: a data
   layer (values, codecs, dirty, validation, submission diff) and an interaction layer
   (touched, error timing, debounce, races, array keys, submit lifecycle). Our Client IS
   the data layer — the heavy half, already built. Riding TanStack means two stores of the
   same truth with per-field sync adapters between them; every sync boundary breeds bugs,
   and programmatic draft writes are invisible to it. Instead: **`FormSession`** in
   `@active-drizzle/react` wraps a Client with `{ touched, submitAttempted, isSubmitting,
   per-field listener map }`; each `Present` subscribes to its own field via
   `useSyncExternalStore` (keystroke re-renders one field). Client stays pure/isomorphic.
   We ride TanStack's *test suite*, not its runtime — the conformance matrix below is the
   certainty contract.
4. **`edit` is never inferred; `view` may default.** (RFC hard wall #2 — keep.)
5. **Meta is semantic only** — no layout/section/grid/step info on fields. (RFC wall #6.)
6. **Role/identity in model predicates = codegen error.** Roles live on controllers, full stop.
7. **`Attr.state` assignment stays legal, transition legality enforced at save-validate** —
   keeps the Attr contract (assign anything, validate on save) instead of throwing on set.
8. **Meta is OPEN, not enumerated (2026-07-18).** Core keys AD understands + arbitrary
   app keys (`icon`, `regulatoryDisclosure`, …). Two rules: (a) statically extractable —
   literals/nested data/analyzable pure fns — or build error; (b) typed via declaration
   merging on `AttrMeta`, so custom keys flow into `FieldMeta`, presenter props, and
   `requires` gates with full checking.
9. **Commit policy (2026-07-18).** Presenters declare their natural commit moment at
   registration (discrete inputs → on change; continuous → blur/debounce); context decides
   what commit DOES: `<Form>` = stage into draft (batch submit) · `<Form autosave>` =
   single-field PATCH per commit · standalone field = autosave by definition (handle owns
   id + mutation, zero props). `bind.state: idle|staged|saving|saved|error` lets any
   presenter render spinners/checkmarks. Optimistic write + rollback-on-failure + version
   echo (409 → refetch + notify) live in FormSession, written once — presenters stay ~30
   lines. The magic toggle: `<loan.isPublished edit />` = flip → optimistic → PATCH →
   rollback on fail, all from policy, none from the presenter.
10. **Headless forever.** AD ships the socket (contract, registry, binding, test-kit,
    one reference presenter as adaptation template), never the bulbs. No shipped UI, no
    CSS story, no silent default presenters.

## The silent-refusal fix (Sat, session 6)

Bug report: add an empty (invalid) note → Save → nothing happens, nothing
saves, nothing shows. Root cause: child errors GATED the parent submit, but
error VISIBILITY is per-session (`touched ∪ submitAttempted`) and the parent
never told its children a submit was attempted. Fixed at the framework level,
plus two adjacent holes the fix exposed in the same gate:

1. `markSubmitAttempted()` now propagates recursively (session → nested
   managers → child sessions → their managers…). A blocked save is never
   silent again — every invalid row lights up.
2. The gate now uses `gateErrors()` (client validate + nested, recursive)
   instead of the flat `allErrors()`: an invalid GRANDCHILD previously
   didn't block the submit at all.
3. `allErrors()` included stale child server errors in the gate — a 422
   routed to a child would wedge resubmits forever even after the fix.
   Server errors are display-only in the gate now, and `resetBaseline()`
   clears them on post-save settle.

Four new regression tests (untouched-child visibility, grandchild gate +
visibility, no-wedge resubmit); react suite 76/76. Browser-verified in
Dealdesk: empty note + Save → "can't be blank" appears instantly, zero
network traffic; fill it in → one PATCH → saved.

## "Are you 100% sure nested-nested works?" — the honest audit (Sat, session 6b)

No — auditing found five more holes, now all fixed, tested, and browser-proven:

1. **Settle didn't recurse.** `commitBaselines` never settled a child's own
   nested managers: a saved new GRANDCHILD stayed `isNew` and was re-created
   by every subsequent save. Now every child matches its saved server row
   (persisted by id, new in order) and hands it down (`settleNested`).
2. **Grandchild 422s dead-ended.** `applyExternalErrors` merged
   `liens[new:1].holder` as a flat key on the middle session. It now routes
   through `refieldErrors` → onward to the grandchild, visible.
3. **Rehydrate ignored children.** `applyEnvelope` now syncs a CLEAN manager
   to the server's rows (refetch picks up children changed elsewhere) —
   never touching a manager with unsaved local edits.
4. **THE DEMO BUG: nested writes weren't permit-governed anywhere.**
   The demo permit never included `notesAttributes` → the server silently
   stripped every note write (my earlier "verification" checked a 200, not
   persistence). Fixed end-to-end as a first-class concept:
   - permit takes `'<assoc>Attributes'` entries (Rails strong-params style)
   - envelope abilities carry `'<assoc>Attributes': 'edit' | 'view'` from
     the same resolved permit → the client LOCKS the array on 'view'
     (Add/Remove hidden, rows render view presenters, payload omits it)
   - create reports forbidden issues like update always did
   - the client surfaces ok-with-issues loudly (base error + console.warn)
     — a permit/UI mismatch can never hide again
   - codegen refuses to emit a nested form a static permit provably never
     accepts (build warning), emits when any permit is dynamic
   - `${Model}Write` types `<assoc>Attributes` as a rows array, not a Pick
5. **Save responses didn't echo includes.** defaultUpdate/defaultCreate
   serialized the bare record — nested children came back WITHOUT ids, the
   client could never settle new rows, and a double-save DUPLICATED notes
   (reproduced live: notes id:3+id:4 with identical bodies). Envelopes now
   reload with `get.include`; a dev tripwire warns when new rows exist but
   the response omits the association (this tripwire caught the live dup).

Suites: react 84, controller 106, core codegen 191 — green. Browser-proven
on Dealdesk: double-save → exactly ONE note in the DB; empty note → visible
"can't be blank"; submit as member → whole form INCLUDING notes self-locks
(admin keeps editing). Docs updated (abilities.md wire contract: nested
abilities entries, save-response includes, issues on create; forms-surface
nested section).

## Security sweep: model-allows vs controller-gates (Sat, session 6c)

Reviewer flagged the class: a MODEL-level allowance (`acceptsNested`) that
lacks a CONTROLLER-level gate. Swept the whole nested + write + read surface
and hardened six vectors — every one verified live against the demo API:

1. **Cross-record ownership (the headline).** `_processNestedAttributes`
   blindly `find(id)`'d and updated/destroyed any child by id. Forging
   another record's child id could re-parent, overwrite, or destroy it.
   Now every id-bearing nested item must ALREADY be a child of this record
   (fk match) or the save fails 422 `row N is not part of this record's X`;
   a nonexistent id fails identically (no id probing). Live: hijack of
   deal-1's note from deal-2 → 422, note untouched.
2. **Parent fk forced.** The owner fk strips from every child row (was
   already set server-side, but accepting it from the wire allowed
   re-parenting). Live: forged `dealId:999` → row saved with correct fk.
3. **STI forge + server-owned fields.** `type` (STI discriminator) and
   timestamps strip from child rows — a client can't forge a subclass or
   backdate through nesting. Live: `type:'Evil'`/`createdAt` never landed.
4. **Destroy is a Rails-style opt-in.** `acceptsNested: true` = create+update
   only; `_destroy` is ignored. Destroying persisted children needs
   `acceptsNested: { allowDestroy: true }`. The generated form hides Remove
   on persisted rows when off; the client never sends `_destroy`. Live:
   legit owned destroy (opt-in on) works.
5. **`_event` is a strict allowlist.** It could name ANY zero-arg record
   method (`_event: 'destroy'`). Now it must be a declared Attr.state
   transition or 400. Live: `_event:'destroy'` → 400, deal survives.
6. **Undeclared grandchildren drop.** A nested `<x>Attributes` in a child
   row is honored only when the CHILD model declares acceptsNested for it —
   otherwise it never reaches an INSERT (was: unknown-column 500 / probing).

Plus a read-side gate: **`expose` now governs `index`** too (a list could
leak columns the detail envelope hides). Filters/sort/ids were already
allowlisted.

New helper `resolveNestedAssociations` (core, duck-typed) + controller-side
`sanitizeNestedWrites` recurse the strip through grandchildren. `NestedAttributesError`
→ 422. Meta carries `allowDestroy`; child meta omits the parent fk and `type`.
Suites: core 879 (2 fixtures updated for the destroy opt-in), react 84+ (24
nested), controller abilities 18 incl. new sanitize/allowlist/index tests.

## Infinite-depth nesting + instant nested writes (Sat, session 7)

Answered "does it support infinite depth?" — it does now, at every layer:

1. **Codegen recursion.** `renderNestedMetaEntry` recurses into a child
   model's own acceptsNested associations, emitting grandchild `kind: 'nested'`
   entries inside the child's `fields`, to arbitrary depth (cycle-guarded by
   table name; `needsValidates` import check walks the whole tree via
   `collectNestedModelsDeep`). Was one level; now unbounded.
2. **Eager-load normalization.** `Relation.includes({ notes: ['reactions'] })`
   lowers ergonomic forms to drizzle's nested `with` shape (`_toDrizzleWith`),
   so a deal envelope can carry `deal → notes → reactions` three deep. The
   controller `only`-serialization takes top-level names (`topLevelIncludeNames`)
   and `IncludeSpec` widens the config type.
3. **Instant nested writes.** `acceptsNested: { instant: true }` makes a nested
   row commit IMMEDIATELY (optimistic, rolled back on failure) via the child's
   own controller WHEN the parent row is persisted; on a brand-new parent it
   stages and rides the parent save, becoming instant once settle gives the
   row an id. NestedArrayManager gained `add`/`patch`/`remove`/`move` instant
   paths + `rows()`; the handle exposes `rows/patch/remove/instant/use`;
   codegen emits `instant/resource/foreignKey` in meta + a `nestedTransports`
   map wired to `client.<resource>.*`.

**Demo: reactions (deal → note → reaction), like/dislike per user.**
Reaction model + ReactionController (autoSet userId, scopeBy own reactions).
Note.reactions is `acceptsNested: { allowDestroy: true, instant: true }`. The
whole feature is ~50 lines of demo code (a ReactionBar widget reading `rows`
and calling add/patch/remove) over the framework machinery.

Verified live against the API:
- 3-level nested READ: deal → notes → reactions load nested.
- 3-level staged WRITE: create deal + note + reaction in one save; envelope
  echoes the whole tree WITH ids (settle works at depth 3).
- Instant CRUD: reaction create autoSets userId; a cross-user update/destroy
  404s (scopeBy ownership); own-reaction update works.
Framework suites: core 880, react 91 (incl. 6 instant-nested tests), controller
abilities 18. Browser click-through pending (Browser MCP was disconnected this
session) — API-level and unit proof stand in.

**Known demo limitation:** the STAGED reaction path (new note) trusts the
client-sent `userId`; the INSTANT path autoSets it server-side. Forcing it in
the staged/nested path needs nested autoSet (a framework feature not yet
built). Through the UI you can only ever be yourself; the raw-API forge is a
demo-acceptable gap, noted for follow-up.
